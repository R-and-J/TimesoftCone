## 1. 역할 분담 원칙

2인 팀이므로 **"주담당 + 부담당"** 체계로 운영 — 주담당자가 진행하되 부담당자가 리뷰/백업.

### 분담 옵션

#### 옵션 A — 도메인 분할 (권장)
각자 전체 스택을 다루되 담당 도메인을 나눔.

| 담당자 | 주담당 도메인 |
|---|---|
| **김기철** | Auction (입찰/낙찰) + HR 연동 + 배치 |
| **오지석** | User/Auth + LeaveBalance + Dividend + Admin |

**장점**: 각자 풀스택 경험, 도메인 전문성
**단점**: 초기 설정 중복 가능

#### 옵션 B — 계층 분할
백엔드/프론트 분리.

| 담당자 | 주담당 |
|---|---|
| **김기철** | 백엔드 전체 |
| **오지석** | 프론트엔드 전체 |

**장점**: 병렬 개발 용이, 명확한 경계
**단점**: 풀스택 경험 부족, API 연계에서 마찰

#### 옵션 C — 혼합 (현실적)
초기 설정은 공동, 이후 도메인 분할 + 프론트는 주로 1인.

→ **권장**: 옵션 A 기반 + 프론트엔드 화면별 1명 주도

## 3. 의사결정 규칙

| 상황 | 결정 방식 |
|---|---|
| 기술 선택 (언어/라이브러리) | **양자 합의** 필수 |
| ADR 신규 추가 | **양자 합의** + 문서화 |
| 마이너 구현 디테일 | 담당자 재량 (PR에서 공유) |
| 의견 불일치 지속 | 외부 레퍼런스 2개 이상 찾아서 비교 → 그래도 안 되면 분반씩 양보 |

## 4. 커뮤니케이션 채널

| 채널 | 용도 |
|---|---|
| GitHub PR | 기술 논의 (우선) |
| GitHub Issues | 할 일 추적 |
| (이 프로젝트 문서 폴더) | 영구 자산 |
| 카카오톡 / 디스코드 | 긴급 의사소통 |
| 오프라인 | 주간 스탠드업 30분 |
