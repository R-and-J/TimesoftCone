# ADR-007: 경매 단위를 "1일권"으로 고정

- **상태**: 🟡 Proposed
- **결정일**: _미결_
- **결정자**: 타임소프트콘

## 컨텍스트

경매의 최소 단위를 어떻게 정의할지 결정 필요. "3일 연속 휴가권" 같은 묶음 상품 허용 여부.

## 초기 고민 (Rejected)

> **옵션 — 가변 단위 허용**: "직원이 3일 연속 휴가가 필요하면 '3일권'을 한 번에 경매에 올린다."

## 회피한 리스크

💣 **설계 복잡도 폭증**

- 매물 크기(일수)에 따라 입찰 가격이 **비선형적**으로 결정됨 → 가격 발견 어려움
- 낙찰 후 부분 사용(1일만 쓰고 2일 남김) 시 **잔여 일수 환불 로직** 필요
- 유찰된 "3일권"을 분해하여 재판매할지 여부 정책 필요
- HR API가 "연속 연차"를 구분해서 받아야 하는지 불명확

**학생 프로젝트 스코프에서 과도한 복잡도**.

## 결정

**경매 매물 = "연차 1일권" 단일 단위**로 고정.

- 3일이 필요한 직원은 **3번 각각 입찰** → 3건 낙찰
- 매물은 모두 동일 속성이므로 시장 가격이 수렴하기 쉬움
- HR API 호출도 항상 `amount: 1` 고정

### SRS 참조
> "본 시스템은 특정 날짜가 아닌 '연차 사용 권리(개수)'를 거래 대상으로 삼으며" (SRS 1.2)

## 결과 및 트레이드오프

### ✅ 긍정적 결과
- **모든 매물 동질** → 시장 가격 발견 용이
- **HR API 호출 단순** (`amount: 1` 고정)
- **부분 환불/분해 로직 불필요**
- **테스트/운영 부담 최소**

### ⚠️ 트레이드오프
- **묶음 할인 불가** — "3일권을 3일 각각 입찰하는 것보다 싸게 제공"하는 번들링 전략 사용 불가
- **장기 휴가 희망자의 UX 불편** — 3일 연속 휴가 원하면 3번 입찰해야 함 + 각 경매 낙찰 보장 없음
- **가격 상승 위험** — 매물 수량이 충분하지 않으면 단일 경매 가격이 치솟을 가능성

### 🛡️ 제약
- 향후 "N일권" 도입 시 본 ADR을 **Superseded by**로 마킹 + 스키마 마이그레이션 필요
- 현재 설계의 **모든 가격 정책은 "1일당"** 기준
