# ADR-018: 경매 정산 규칙 (패자 환불 · 입찰 취소)

- **상태**: ✅ Accepted
- **결정일**: 2026-05-14
- **결정자**: 타임소프트콘

## 컨텍스트

[[ADR-009]]는 "입찰 시 홀드 없이 즉시 차감, 유찰/취소 시 REFUND"라고만 명시했다. 그러나 경매의 가장 기본 메커니즘인 **"상위 입찰에 밀린 패자를 *언제* 환불하는가"**가 어디에도 정의되지 않았다. 또한 **입찰자가 자기 입찰을 취소할 수 있는가**도 미정이었다.

이 둘은 구현 디테일이 아니라 *기획 결정*이다 — 잘못 정하면 유동성 경색·경쟁 교란·에스크로 정합성 붕괴로 이어진다.

## 결정 1 — 패자 환불: 밀리는 즉시 환불

상위 입찰이 접수되는 *그 트랜잭션 안에서* 직전 최고가 입찰자에게 즉시 환불한다.

### 후보 비교

| 기준 | 옵션 A: 밀리는 즉시 환불 ✅ | 옵션 B: 마감 시 일괄 환불 |
|---|---|---|
| 유동성 | 패자 콘 즉시 회수 → 다른 경매 입찰 가능 | 경매 기간(3일) 내내 패자 콘 묶임 |
| 환불 트랜잭션 수 | 입찰마다 1건 (많음) | 마감 시 1회 일괄 (적음) |
| 사용자 직관성 | 높음 — "밀렸으니 돌려받음" | 낮음 — "왜 내 콘이 묶여 있지?" |
| 마감 직전 입찰 | 자유 (콘 회수돼 있으므로) | 경색 — 분산 오픈이라 동시 진행 경매가 많은데 입찰 불가 |
| 구현 복잡도 | 중 (입찰 트랜잭션에 환불 로직 포함) | 중 (마감 배치에 일괄 환불) |

### 채택 이유

1. **유동성 경색 회피** — 분산 오픈([[business-rules]] OP-4)으로 여러 경매가 동시 진행되는데, 옵션 B는 한 경매에 입찰했다 밀리면 3일간 다른 경매 입찰 불가
2. **사용자 직관** — "상위 입찰 발생 → 즉시 환불 알림"이 자연스러움
3. **[[ADR-009]] 정합** — "홀드 없이 즉시 차감, 취소 시 REFUND"의 자연스러운 연장

### 흐름

```
입찰 접수 (단일 DB 트랜잭션, SQLite write 락(lockAuction) 보유 중):
  ├─ auction.placeBid() — OpenState, amount >= highest_bid + 증분 검증
  ├─ BiddingCurrency.debit(신규입찰자, amount)
  ├─ LEDGER_ENTRY INSERT (BID, -amount, 신규입찰자)
  ├─ if 직전 최고가 입찰자 존재:
  │     BiddingCurrency.credit(직전입찰자, 직전금액)
  │     LEDGER_ENTRY INSERT (REFUND, +직전금액, 직전입찰자)
  ├─ auctions.highest_bid / winner_id 갱신
  └─ COMMIT
```

→ 경매 마감 시점에는 **최종 낙찰자만 차감 상태**. 마감 시 `WIN`으로 확정, 에스크로 귀속.

### 에스크로 정합성

`Σ(BID + WIN) − Σ(REFUND + DIVIDEND) = ESCROW.balance` (currency별, [[db-schema]] DB-RULE-4)

- 입찰 시 BID(-) 누적, 밀리면 REFUND(+)로 상쇄 → 진행 중 경매의 패자 입찰분은 자동 정리
- 마감 시 낙찰자 입찰분만 남아 에스크로 잔액으로 확정

## 결정 2 — 입찰 취소: 불가

한번 제출한 입찰은 입찰자가 스스로 취소할 수 없다.

### 채택 이유

1. **경쟁 교란 방지** — 취소가 가능하면 "입찰 → 경쟁자 떠보기 → 취소" 같은 어뷰징 가능
2. **마감 직전 스나이핑 방지** — "취소 치고 빠지기" 차단
3. **구현 단순** — 취소 가능 시점·조건·환불 처리 로직 불필요
4. **즉시 환불과 결합** — 어차피 상위 입찰에 밀리면 즉시 환불되므로, 입찰자가 "묶인 콘을 빼려고" 취소할 동기 자체가 약함

### 거절한 대안

- **최고가가 아닐 때만 취소 가능**: "밀리는 즉시 환불"과 결합하면 *이미 환불된 상태*라 취소할 게 없음 → 무의미
- **마감 직전 N분 제외하고 자유 취소**: 스나이핑 방지 로직이 추가로 필요 → 복잡도 대비 이득 없음

## 결과 및 트레이드오프

### ✅ 긍정적 결과

- **유동성 경색 해소** — 분산 오픈 환경에서 패자가 즉시 다음 경매로 이동 가능
- **경쟁 무결성** — 입찰 취소 불가로 떠보기·스나이핑 차단
- **에스크로 자동 정합** — 진행 중 패자 입찰분이 REFUND로 자동 상쇄
- **구현 경계 명확** — 환불은 *입찰 트랜잭션 내부*, 마감 배치는 낙찰 확정만

### ⚠️ 트레이드오프

- **환불 트랜잭션 수 증가** — 입찰 1건마다 (최대) REFUND 1건. 단 모두 `LEDGER_ENTRY` INSERT라 부담 작음
- **입찰 신중성 요구** — 취소 불가이므로 UI에서 입찰 확정 전 명확한 확인 단계 필요
- **재입찰 시 차감-환불-차감 반복** — 같은 사람이 연속 재입찰하면 ledger에 BID/REFUND가 여러 건. 감사 시 정상 패턴으로 인지해야 함

### 🛡️ 제약

- 패자 환불은 **반드시 입찰 트랜잭션과 동일 트랜잭션** — 별도 비동기 처리 금지 (정합성)
- 입찰 UI는 **확정 전 확인 모달 필수** — 취소 불가 명시
- 퇴사자가 최고가인 경매는 환불 규칙과 별개로 처리 — [[edge-cases]] EC-1
