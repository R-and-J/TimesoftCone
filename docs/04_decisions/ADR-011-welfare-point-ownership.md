# ADR-011: 복지 콘 시스템 자체 보유 (Wallet Master Ownership)

- **상태**: ✅ Accepted
- **결정일**: 2026-05-14
- **결정자**: 타임소프트콘
- **관련**: [[ADR-009]] 보완 (전제 조건 충족 불가 상황 대응)

## 컨텍스트

[[ADR-009]]는 "이미 운영 중인 사내 복지 콘 시스템을 재활용"한다는 *전제* 위에 결정되었다. 그러나 본 프로젝트의 대상 사내 환경을 검토한 결과:

- **소프트웨어 형태의 복지 콘 시스템이 존재하지 않음** — HR에서 수기·엑셀로 관리
- 따라서 ADR-009의 전제 조건 1·2·3이 모두 충족 불가:
  - 전제 1 ❌ — 시스템 부재
  - 전제 2 ❌ — `복지 콘 → 복지카드 한도` 변환 API 부재
  - 전제 3 ❌ — *발행* 주체가 시스템 차원에서 부재

→ ADR-009의 "개발 스코프 축소" 정당성이 부분 붕괴. 누군가는 발행·적립·만료를 책임져야 한다.

또한 현재 [[db-schema]]가 이미 `USERS.current_point INT` 컬럼을 *본 시스템 DB가 직접 보유*하고 있어, 사실상 마스터 역할을 하고 있다 (ADR-009 전제 3과 모순).

## 초기 아이디어 (Rejected)

### 옵션 A — ADR-009 유지, "전제가 충족된다고 가정"
- 학교 프로젝트라서 *가정*으로 진행하고 실 배포는 다른 사람 몫
- **거절 이유**: 발표·평가에서 "전제가 깨졌는데도 진행" 지적 시 방어 불가. ADR-009와 스키마의 모순 미해소.

### 옵션 B — 신규 화폐 발행 (LEAVE COIN)
- ADR-009의 옵션 B로 회귀
- **거절 이유**: ADR-009 §회피 리스크 ("유가증권 취급 위험", 세무 부담) 그대로 부활. 학교 프로젝트의 *법적 자문* 부담.

### 옵션 D — Mock HR + 외부 시스템 *가정*만 유지
- 학교 프로젝트는 시뮬레이션, 실 도입은 별도 프로젝트
- **부분 채택**: 학교 발표 관점에서는 유효. 그러나 *코드 산출물*은 무엇을 만들지 모호 → 본 결정과 결합 필요.

## 회피한 리스크

💣 **ADR-009 전제 붕괴로 인한 시스템 *공중 분해***
- 잔액의 마스터가 어디인지 불분명 → 동기화 정책 부재
- 발행 주체 부재 → 시드 데이터·관리자 적립이 *어디 명세*에도 없음

💣 **스코프 회색지대**
- "외부 시스템이 알아서 해줄 것"이라는 모호한 위임
- 실제 개발 단계에서 *경계가 무너져* 본 시스템이 점점 더 많은 책임을 가져가게 됨

## 결정

**본 시스템이 복지 콘의 *잔액 마스터*를 보유하되, 스코프는 명시적으로 한정한다.**

### 책임 범위 (In Scope)

| 책임 | 구현 |
|---|---|
| **잔액 보유** | `wallet(user_id, currency, balance)` 테이블이 마스터 |
| **차감 (입찰)** | `BiddingCurrency.debit()` — ADR-010 인터페이스 |
| **적립 (환불·관리자 지급)** | `BiddingCurrency.credit()` + 관리자 적립 API |
| **로그** | `LEDGER_ENTRY` 테이블에 모든 변동 INSERT (ADR-009 §대장 불변) |
| **만료 정책** | 본 시스템의 wallet 잔액은 *만료 없음* (단순화). 미사용 콘은 계속 보유 |
| **시드 (초기 잔액)** | 운영 시 마이그레이션 도구로 HR 엑셀 → wallet 일괄 INSERT |

### 비책임 범위 (Out of Scope)

| 항목 | 처리 |
|---|---|
| 헬스장·식대·도서 등 *외부 사용처* | 본 프로젝트 스코프 외. 미래에 `ExternalUsageAdapter`로 통합 (어댑터 추가만, 코어 무수정) |
| 분기/연간 *자동 적립* (인사 정책) | 운영자가 관리자 API로 일괄 적립. 자동화는 본 프로젝트 외 |
| 사내 복지 *카드 한도* 자체 관리 | HR 시스템 책임. 본 시스템은 `/api/hr/welfare`로 *통보*만 |

### 신규 API (SRS FR-5.x로 명세 예정)

```http
POST /api/admin/wallet/credit
{
  "userId": "...",
  "amount": 50000,
  "currency": "WELFARE_POINT",
  "reason": "2026 Q2 분기 복지 콘 지급"
}
```

- 권한: `ADMIN` 전용 (RBAC)
- `LEDGER_ENTRY`에 `action_type = 'CREDIT_ADMIN'` 으로 INSERT
- 감사용 `reason` 필수

### Currency Abstraction과의 정합

[[ADR-010]]의 `BiddingCurrency` 구현체 `WelfarePointProvider`는 **본 시스템 내부의 wallet 테이블만 조작**한다. 외부 HR 호출 없음.

```
[옵션 B 채택 시 — 본 ADR]
  AuctionService
       │
       ▼
  BiddingCurrency (interface)
       │
       ▼
  WelfarePointProvider
       │
       ▼
  wallet 테이블 (본 시스템 DB 마스터)
```

→ 입찰 결제 경로에서 외부 시스템 호출이 *0건*. [[ADR-005]]의 Outbox 경계는 *배당 출금* 시점에만 적용된다.

## 결과 및 트레이드오프

### ✅ 긍정적 결과

- **마스터·동기화 모호성 해소** — 본 시스템이 단일 진실 공급원(Single Source of Truth)
- **ADR-009와 스키마의 모순 해소** — `USERS.current_point` 컬럼이 wallet 테이블로 정상화
- **입찰 경로의 외부 의존 0** — 사용자가 원하던 "결제 완전 독립" 의도 충족 (앞선 논의 §2)
- **테스트 단순화** — Mock HR이 *배당 채널*에만 필요 (입찰엔 불필요)

### ⚠️ 트레이드오프

- **스코프 증가** — 관리자 적립 API, 마이그레이션 도구, 잔액 조회 UI 추가 ([[wbs]] 갱신 필요)
- **운영 부담** — 분기·연간 일괄 적립은 운영자 수동 작업
- **ADR-009 §개발 스코프 축소 일부 무효** — 발행 로직은 *없지만* 적립 로직은 새로 필요

### 🛡️ 제약

- 외부 사용처가 *미래에* 추가될 때까지, 본 시스템 wallet의 잔액은 *외부와 동기화되지 않는다*
- 헬스장 등 외부 사용처가 *동시 운영 중*이라면, 사용자가 본 시스템의 잔액과 사내 다른 곳의 잔액이 일치하지 않을 수 있음 → 이는 *현실 제약*으로 받아들이고, 사용자 UI에 명확히 표기
- 관리자 적립 권한은 RBAC + 감사 로그로 엄격히 통제 — `CREDIT_ADMIN` 액션은 `reason` 필수
