# ADR-023: 자립형 배포 — 내부 포인트 소모처 (소비채널)

- **상태**: 🟡 Proposed (의도/방향 정리, 구현은 후속 PR)
- **결정일**: 2026-05-30
- **결정자**: 타임소프트콘

## 컨텍스트

[[ADR-009]](v2, 포인트 재활용)와 [[ADR-021]](이식형 핸드오프 export)에서 정리한 대로, 우리 시스템의 *재화*는 도입 회사의 **기존 복지포인트**를 재활용하고, 소비는 회사의 기존 복지몰/제휴몰에서 일어난다고 가정한다.

[[ADR-022]](배포 모드)는 *신원/인증* 측면을 **위임형(ezpass 등)** / **자립형(LocalAuthProvider)** 두 모드로 일반화했다. 즉 "그룹웨어가 있으면 위임, 없으면 자립" 패턴. 하지만 *재화* 측면은 이 일반화가 안 돼 있다 — 자립형 회사라도 **외부 복지포인트 플랫폼이 있다고 암묵적으로 가정**하고 있다.

문제:
- 복지포인트 시스템이 **없는** 회사도 도입 대상이다(우리는 이식형 B2E 플랫폼).
- 자립형 배포에서 우리 `wallet`은 잔액 정본([[ADR-011]])이지만, **포인트의 실가치(소비할 곳)** 가 없으면 경매·배당이 동기를 잃는다 — escrow는 그대로 쌓이는데 사용자가 받을 게 없음.

→ 자립형 배포가 진짜로 "그룹웨어 0으로도 돈다"가 성립하려면, **소비처(redemption channels)** 도 같이 자립해야 한다. 이미지 메모(2026-05-30): "AI 구독권(3만/30만), 식권, 카페 상품권, 카카오톡 선물하기" 등의 상품을 우리 시스템 안에 두자.

## 결정

**자립형 배포에 한해, 우리 시스템이 내부 소비처를 제공한다 — 상품 카탈로그 + 교환(REDEEM) use case + 발송 어댑터(들).**

### 운용 예시 (상품 카탈로그)

| 상품군 | 예시 |
|---|---|
| 디지털 구독 | ChatGPT/Claude Pro 1개월(₩30,000), 1년(₩300,000) |
| 식권 | 사내 식당·제휴 식권 |
| 카페 상품권 | 스타벅스·투썸 등 |
| 카카오톡 선물하기 | 외부 API 연동 시 즉시 발송 가능 |

가격은 회사 정책에 따라 운영자가 설정. 재고/유효기간 관리.

### 구조 — `RedemptionChannel` 포트 + 어댑터 (ADR-010 패턴)

[[ADR-010]](Currency 추상화)이 입찰을 `BiddingCurrency`로, [[ADR-008]] 배당을 `PayoutChannel`로 분리한 것과 **같은 패턴**으로 `RedemptionChannel` 포트를 신설:

| 어댑터 | 동작 | 상태 |
|---|---|---|
| `InternalCatalogRedemption` | 우리 `redemption_item` 카탈로그에서 재고 차감 + 사용자 wallet 차감 + `REDEEM` ledger INSERT (인증쿠폰/상품권 코드 발급은 본 시스템이 보관) | 자립형 기본 |
| `KakaoGiftRedemption` *(예시)* | 카카오 선물하기 API로 즉시 발송 — **외부 호출이므로 [[ADR-005]] Outbox 경유** ([[ADR-013]] 역할 분리 유지) | 선택 |
| `WelfareMallRedemption` *(미구현)* | 회사 기존 복지몰로 위임 | 위임형 배포용 향후 |

배포 모드별:
- **위임형(기본)**: 소비는 회사 기존 복지몰 → 우리는 소비처 미제공(현 상태 유지).
- **자립형**: `InternalCatalogRedemption`을 기본으로, 외부 기프트 API 어댑터를 선택적으로 활성.

### 도메인 추가(예정)

- **상품**: `RedemptionItem(id, sku, name, priceP, stock, active, …)`
- **교환**: `RedemptionOrder(id, userId, itemId, priceP, status, deliveryRef, createdAt)` — `PENDING → FULFILLED|FAILED|REFUNDED` 상태 전이
- **원장**: `LedgerActionType`에 `REDEEM`(차감, 음수) / `REDEEM_REFUND`(환불, 양수) 추가. CHECK 제약 갱신.

## 인바리언트와의 정합

- **DB-RULE-1 Insert-Only ledger** 유지: 교환은 `REDEEM` ledger INSERT, 정정은 `REDEEM_REFUND` INSERT(=새 행). UPDATE/DELETE 금지 그대로.
- **NFR-2 에스크로 등식**: 교환은 escrow 흐름과 *무관* — `CREDIT_ADMIN`과 같은 분류(escrow 등식에서 제외). 사용자 잔액 차감이지 경매금 흐름이 아님.
- **#1 회사 예산 0원 선투입**: 자립형에서도 *선투입은 0*. 회사가 부담하는 것은 **교환이 일어난 시점의 상품 원가**(후정산). escrow 모델은 그대로.
- **#5 P2P 직거래 금지**: 교환은 사용자↔시스템 1:1. 사용자간 양도 없음.
- **#7 Hexagonal** ([[ADR-012]]): `RedemptionChannel` outbound port, Use Case는 채널 구현체를 모름.
- **#6 단일 트랜잭션**: 교환 1건(재고 차감 + wallet 차감 + ledger INSERT + RedemptionOrder INSERT)은 한 tx. 외부 기프트 API 호출이 필요한 채널은 [[ADR-005]] Outbox로 분리(같은 tx 안에 outbox 적재).

## 결과 및 트레이드오프

### ✅ 긍정적
- **자립형 배포 완결**: ADR-022(신원) · [[ADR-016]](연차) · [[ADR-011]](지갑) · **본 ADR(소비처)** 로 그룹웨어/복지몰 0인 회사도 도입 가능.
- **포트 일관성**: `BiddingCurrency` · `PayoutChannel` · `RedemptionChannel` — 세 포트 모두 동일 패턴, 도메인은 셋 다 모름.
- **수익 옵션 열림**: 외부 기프트 API 마진/수수료 등 비즈니스 옵션이 생김(범위 외, 별도 ADR).

### ⚠️ 트레이드오프
- **운영 부담 증가**: 카탈로그 관리·재고·제휴사 연동·정산 — 위임형(회사 복지몰)이 알아서 해주던 것을 *우리*가 한다.
- **외부 기프트 API = 실 결제**: 카카오 선물하기 등 호출 시점에 회사 예산 직접 차감(escrow 외 경로). 1인 한도/주간 캡 같은 정책 변수 필요.
- **법/세제**: 식권·상품권 지급은 근로소득 과세 이슈 발생 가능 — 도입사 인사/세무와 사전 협의.
- **재고 0 / 발송 실패 처리**: 동기 발송 실패 시 자동 환불(`REDEEM_REFUND`) 흐름 설계 필요.

### 🛡 제약
- 본 ADR은 **자립형 배포에 한정**(위임형은 그대로 회사 복지몰).
- 별도 마이그레이션 + 새 도메인(`Redemption`) + 새 ledger 액션타입(`REDEEM`/`REDEEM_REFUND`) — 후속 PR.
- 외부 기프트 API는 [[ADR-005]] Outbox로만(인바리언트 #6 유지).
- 운영 파라미터(1인 한도·재고 상한 등) 후속 결정.
