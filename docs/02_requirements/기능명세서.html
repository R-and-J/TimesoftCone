기능 명세서

타임소프트콘 - 연차 경매 시스템 (B2E 연차 경매·에스크로 배당 플랫폼) · 김기철, 오지석

목적 · 요구사항 설계서에서 분류한 각 기능의 상세 동작(행위자·사전조건·처리 흐름·입출력·화면·API·예외)을 명세한다. “무엇을”은 SRS, “테스트로 무엇이 done인가”는 인수 조건, “왜 그렇게”는 ADR에 위임한다.

  2. 공용 풀 수집·자산화 (LeavePool)

  5. 낙찰·에스크로·휴가 부여 (Settlement)

  7. 연말 배당 정산 (Dividend)

  11. 화면-기능 매핑
  12. API 일람

표기 규약

각 기능은 행위자 · 사전조건 · 처리 흐름(트랜잭션 경계 명시) · 입력/출력 · API/화면 · 예외 · 불변식 항목으로 명세한다. 400 CODE 형식은 오류 응답 코드이다. ★ 표시는 핵심 기능.

1. 인증·신원 (Auth)

F-1.1 로그인

행위자
EMPLOYEE / ADMIN
사전조건
유효한 사번/자격 증명
처리 흐름
  1. 클라이언트가 사번·비밀번호를 POST /api/auth/login으로 전송
  2. AuthProvider가 자격 검증 - delegated 모드는 EzpassAuthProvider가 내부 ezpass에 위임, local 모드는 LocalAuthProvider가 bcrypt 해시 대조 (ADR-022)
  3. 성공 시 신원(이름/부서/직급/직책/role)을 ezpass에서 동기화 (ADR-020)
  4. sub/role/empId를 담은 JWT 발급
입력
{ empId, password }
출력
{ token, user: { id, name, role, … } }
화면 / API
/login · POST /api/auth/login
예외
자격 불일치 → 401
비고
모드는 AUTH_MODE 환경변수로 선택. 휴가 잔액은 내부 소유이므로 신원만 외부 동기화.

2. 공용 풀 수집·자산화 (LeavePool)

F-2.1 연말 풀 수집 · Stake 생성 - FR-1.1 ★

행위자
SYSTEM(LeavePoolScheduler) / ADMIN(수동 트리거)
사전조건
연말 cutoff(기본 12/31 23:59) 도래, 해당 target_year 미수집
처리 흐름
(단일 tx)
  1. 전 직원의 REGULAR 미사용 연차 일수 집계
  2. 직원별 미사용 N일 → 익년도 “연차 1일권” 매물 N개 생성(초기 상태 CREATED)
  3. stake(user_id, year) 행 생성 - contributed_days, stake_ratio = contributed_days / Σ
  4. leave_pool_run(target_year) 기록 (멱등성)
  5. AuctionInventoryCreatedEvent 발행 → ADMIN 알림
입력 / 출력
{ targetYear }(수동 시) / 생성 매물 수·Stake 요약. 부수효과: 매물·stake·leave_pool_run INSERT
API / 화면
POST /api/admin/leave-pool/collect · 미리보기 GET …/upcoming · /admin/auctions, /admin/ops
예외
미사용 0일 직원 → 스킵(stake row 미생성, AC-1.1.3) · 동일 연도 재실행 → target_year UNIQUE로 멱등 무효화(AC-1.1.4)
불변식
INV-2(REGULAR만 풀 전환), INV-1(원장). 연차 반납 시 콘 선지급 절대 불가.

F-2.2 분산 오픈 (재고 릴리스) - FR-1.1 보조

행위자
SYSTEM(ReleaseInventoryScheduler) / ADMIN
사전조건
CREATED 상태 매물 존재 + release_policy 설정
처리 흐름
release_policy(기본 cadence:none = 즉시 전량, 또는 주기 배포)에 따라 supply → OPEN 전이. 회차마다 정해진 개수만 오픈.
API
POST /api/admin/leave-pool/release · 정책 PATCH /api/admin/release-policy
비고
OP-4/OP-5(분산 오픈)의 구현. 기본값은 즉시 전량 오픈.

3. 경매 매물 관리 (Auction Mgmt)

F-3.1 매물 생성·스케줄

행위자
ADMIN
처리 흐름
관리자가 시작가·기간·최소 증분을 지정해 매물 생성. 시작가는 OP-6 3모드(자유/고정 최소가/작년 평균) 중 선택, 기본 고정 최소가 30,000 P.
입력 / API
{ startPrice, minIncrement, startedAt, endsAt, … } · POST /api/admin/auctions, …/{id}/open, …/schedule, …/next-id
화면 / 예외
/admin/auctions · 일반 직원 →403

F-3.2 매물 운영 (연장·즉시마감·재오픈·취소)

행위자 / 기능
ADMIN · 마감 연장(anti-snipe 수동), 즉시 마감, 유찰 재오픈, 취소
API
…/{id}/extend, …/{id}/close-now, …/{id}/reopen-unsold, POST /api/admin/auctions/cancel

F-3.3 유찰 재고 활용·청산 - FR-4.2

행위자
ADMIN
처리 흐름
  • 유찰(UNSOLD) 매물을 EVENT 휴가로 수동 지급 → leave_balance(EVENT) += 1, status='AWARDED', 에스크로 변동 없음
  • 연말 잔여 재고 영구 삭제(청산) → EXPIRED, 익년 이월 0
입력 / API
{ empId, reason }(reason 필수 - 감사) · POST …/{id}/grant-event, POST …/purge-unsold
예외
일반 직원 →403 · 이미 AWARDED 매물 재지급 → 409 INVALID_STATE_TRANSITION (AC-4.2.E1~2)

4. 실시간 입찰 (Bidding)

F-4.1 입찰 + 즉시 환불 - FR-2.1 ★ 핵심

행위자
EMPLOYEE (ADMIN도 참여 가능 - GOV-1, 단 actor_role 기록)
사전조건
매물 OPEN, 마감 시각 이전, wallet.balance ≥ 입찰가, 입찰가 ≥ 현재가 + 최소 증분
처리 흐름
(단일 DB tx,
PrismaUnitOfWork.run())
  1. lockAuction() - no-op UPDATE로 SQLite write 락 선점(직렬화, CUT-1)
  2. 신규 입찰자 wallet 차감 + LEDGER_ENTRY(BID, -amount) INSERT
  3. 직전 최고가 입찰자 존재 시: 그의 wallet 환급 + LEDGER_ENTRY(REFUND, +직전금액) INSERT (즉시 환불)
  4. auctions.highest_bid/highest_bidder_id 갱신
  5. 커밋 후 BidPlacedEvent 발행 → SSE 브로드캐스트(BID_UPDATED) + 알림(NotificationObserver)
입력 / 출력
POST /api/auctions/{id}/bids { amount } / 200 갱신된 경매 상태. 부수효과: 지갑·원장 2~3행
화면
/auction/detail (SSE 스트림 …/{id}/stream)
예외
400 POINT_INSUFFICIENT 잔액 부족(AC-2.1.E1) · 400 BID_TOO_LOW 현재가+증분 미만(AC-2.1.E2) · 409 AUCTION_CLOSED 마감 경과(AC-2.1.E3)
동시성
동시 입찰은 write 락으로 직렬화 → 뒤 입찰은 갱신된 현재가로 재검증(BID_TOO_LOW 가능, AC-2.1.E4)
불변식
INV-1(원장), INV-6(단일 트랜잭션), INV-4(에스크로 진행 중에도 등식 일관)

※ 자기 자신에 재입찰(현재 최고가가 본인) 시 REFUND+BID가 동시에 원장에 기록된다 - AC-2.1.E5.

5. 낙찰·에스크로·휴가 부여 (Settlement)

F-5.1 낙찰 확정 - FR-2.2 + FR-2.3 ★ 핵심

행위자
SYSTEM(SettleDueAuctionsScheduler, 기본 60초 틱) / ADMIN(수동)
사전조건
매물 마감 시각 도달, 상태 OPEN
처리 흐름
(단일 tx,
all-or-nothing INV-6)
  1. status: OPEN → CLOSED → AWARDED(낙찰자 존재) 또는 UNSOLD(입찰자 없음)
  2. 낙찰 시: LEDGER_ENTRY(WIN, escrow_snapshot) 기록 → 차감분이 에스크로로 귀속(ESCROW.balance += highest_bid)
  3. FR-2.3 휴가 부여: 낙찰자에게 AUCTION 연차 +1 - LeaveGrantPort.grant() → InternalLeaveAdapter가 leave_balance(user, year, AUCTION).allocated_days += 1 UPSERT 같은 트랜잭션 내
  4. 멱등 키 auction-{auctionId}-winner-{userId} - 중복 호출 무효
  5. 커밋 후 AuctionWonEvent 발행 → SSE AWARDED
API / 화면
POST /api/admin/auctions/settle-due(전체), …/{id}/settle(단건), …/open-due · /admin/ops
예외
낙찰자 마감 직전 퇴사 → 휴가 부여 불가 → 유찰(UNSOLD) 처리(AC-2.2.E1, EC-1) · DB tx 실패 → 전체 롤백 + 관리자 Slack Critical 알림 + 수동 감사(AC-2.2.E2)
불변식
INV-6(단일 tx), INV-4(에스크로 천장), INV-2(AUCTION 속성 부여)

6. 휴가 차감 우선순위 (Leave Use)

F-6.1 강제 우선순위 차감 - FR-3.1

행위자
ADMIN 전용 엔드포인트(외부 그룹웨어 승인 트리거를 대리). 기안/승인 워크플로 자체는 범위 외(ADR-016).
사전조건
차감 요청 일수 ≤ 총 잔액
처리 흐름
(planLeaveDeduction) AUCTION → EVENT → REGULAR 순 강제 차감. 1순위 잔고 0이면 다음 순위 폴백. 사용자는 순서 선택 불가(API에 type 인자 없음).
입력 / 출력
POST /api/admin/leave/use { userId, year, days } / 차감 후 속성별 잔액
예외
400 INSUFFICIENT_LEAVE 총 잔액 부족(AC-3.1.E1) · 400 INVALID_AMOUNT 일수 ≤ 0(AC-3.1.E2)
예시
(AUCTION/EVENT/REGULAR) 1/2/15에서 4일 차감 → 0/0/14 (순차 소진, AC-3.1.4)
불변식
INV-3(강제 우선순위), 트랜잭션 단위(부분 차감 후 실패 시 전체 롤백)

7. 연말 배당 정산 (Dividend)

F-7.1 배당 정산 - FR-4.1 ★ 핵심

행위자
SYSTEM(YearEndDividendScheduler, cutoff 게이트) / ADMIN(수동)
사전조건
연말 cutoff 도래, 해당 연도 미정산
처리 흐름
  1. 해당 연도 escrow_balance(통화별 파생) 산정
  2. 직원별 raw = escrow × stake_ratio → floor 내림
  3. remainder = escrow − Σfloor → Stake 1위에게 몰아주기(동률 시 user_id 오름차순)
  4. 직원별 LEDGER_ENTRY(DIVIDEND) INSERT + PayoutChannel.payout() Outbox 발행(복지카드 한도 증액)
입력 / 출력
POST /api/admin/dividend/settle { year } / 직원별 배당액. 불변식: Σ배당 = escrow_balance(통화별), 1콘도 초과 불가
조회 / 화면
직원 본인 GET /api/dividend/me/{userId}?year= · /dividend(지분율·순위·예상 배당), /admin/ops
예외
산정액 불일치 → 배치 중단 + 수동 감사 · 재실행 → 멱등 키로 중복 방지(409, AC-4.1.E2) · 퇴사자 Stake → 산정 포함하되 wallet 소멸 후라 회사 환수(AC-4.1.E1, EC-1)
불변식
INV-4(에스크로 천장), INV-1(원장). CREDIT_ADMIN은 집계 제외.

8. 지갑·원장·관리자 적립 (Wallet/Ledger)

F-8.1 관리자 콘 적립 - FR-5.1

행위자
ADMIN 전용
처리 흐름
직원 wallet에 콘 적립 → LEDGER_ENTRY(CREDIT_ADMIN, +amount, reason) INSERT + WalletCreditedEvent 발행. wallet row 없으면 UPSERT.
입력 / 화면
POST /api/admin/wallet/credit { userId, amount, reason } · /admin/members
예외
403 FORBIDDEN EMPLOYEE 호출(AC-5.1.E1) · 400 REASON_REQUIRED reason 누락/공백(AC-5.1.E2) · 400 INVALID_AMOUNT amount ≤ 0(AC-5.1.E3) · 404 USER_NOT_FOUND(AC-5.1.E4)
불변식
CREDIT_ADMIN은 외부 적립 → 에스크로 등식과 분리 집계(INV-4 예외)

F-8.2 잔액·내역 조회 - FR-5.2

행위자 / 기능
EMPLOYEE(본인) / ADMIN(전체) · 본인 wallet 잔액, 거래 내역(원장 필터·페이지), 휴가 잔여, Stake/예상 배당 조회. 관리자는 전체 원장 조회·필터.
API
GET /api/users/{id}/balance, …/activity, …/leave · ADMIN GET /api/admin/ledger(필터+페이지), GET /api/wallet/{userId}
화면
/activity, /dashboard, /admin/ledger
권한(ABAC)
/users/me/*는 토큰 user_id 일치 검증, 불일치 시 403(AC-5.2.E2). 페이지네이션 기본 20·최대 100, 최신순.
예외
401(토큰 만료/없음), 403(권한)

F-8.3 원장 감사 가시성

기능
모든 거래는 Insert-Only 원장에 영구 기록(INV-1). action_type 필터(BID/REFUND/WIN/DIVIDEND/CREDIT_ADMIN)로 감사 뷰 조회. LEDGER_ENTRY에 UPDATE/DELETE 시도 시 DB 트리거 예외(AC-S.2).
내보내기
GET /api/admin/export(원장), …/export/leave-grants, …/export/dividends, …/export/spending

9. 복지 교환몰 (Redemption)

초기 SRS 이후 개발 중 추가된 기능 - ADR-023. 콘을 사내 복지 품목으로 교환하는 내부 상환 채널. P2P가 아니며 회사가 발행/승인을 중개(B2E 유지).

F-9.1 교환 요청 (직원)

행위자 / 처리
EMPLOYEE · 품목 목록 조회 → 콘 차감 교환 요청 생성(또는 커스텀) → 관리자 승인 대기 → 승인 후 확정/수령
API / 화면
품목 GET /api/redemption/items, 요청 POST /api/redemption/requests[/custom], 수령 …/requests/{id}/confirm, 내 주문 GET /api/users/{userId}/redemption-orders · /redemption

F-9.2 교환 승인 / 품목 관리 (관리자)

행위자 / 기능
ADMIN · 교환 요청 승인/반려, 품목 CRUD·활성화, 변경 감사
API
GET /api/admin/redemption-requests, …/{id}/approve, …/reject, …/summary · 품목 GET/POST /api/admin/redemption/items, …/{id}, …/{id}/active, …/{id}/audits
화면 / 불변식
/admin/redemption, /admin/store · 콘 차감·환급 모두 원장 INSERT(INV-1), P2P 부재(INV-5) 유지

10. 사용자 충전 요청 (Charge Request)

추가 기능 - ADR-024. 직원이 콘 충전을 요청하고 관리자가 승인하는 흐름. 직접 충전이 아니라 승인 게이트를 둔다(감사·COI 통제).

F-10.1 충전 요청 (직원)

행위자 / 처리
EMPLOYEE · 충전 요청 생성 → 관리자 승인 시 wallet 적립(원장 INSERT) / 반려 시 무효
API / 화면
POST /api/wallet/charge-requests · /dashboard//activity 내 진입

F-10.2 충전 요청 승인 (관리자)

행위자 / API
