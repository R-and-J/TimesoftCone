## 1. 서론

### 1.1. 목적

본 문서는 사내 휴가 사용의 양극화 문제(고연차의 잉여 연차와 저연차의 연차 부족)를 해결하기 위해 기획된 **사내 연차 경매 시스템**의 소프트웨어 요구사항을 정의한다.

본 시스템은 **근로기준법(휴식권 매매 금지)을 준수**하고 회사의 재무적 리스크를 완벽히 통제하기 위해 **"수익 공유 및 후 배당(Escrow & Dividend)" 아키텍처**를 채택한다.

본 문서는 개발, QA(테스트) 및 향후 유지보수의 명확한 기준을 제공하는 데 목적이 있다.

### 1.2. 범위

본 시스템은 특정 날짜가 아닌 **"연차 사용 권리(개수)"**를 거래 대상으로 삼으며, 다음의 핵심 범위를 가진다.

- **자산화 시스템**: 미사용 연차 취합 및 기여자 지분(Stake) 설정 (연말 배치)
- **경매 시스템**: 사내 복지 포인트를 재화로 한 "연차 1일권" 실시간 최고가 입찰 및 낙찰
- **에스크로(Escrow) 시스템**: 낙찰 대금을 임시 보관하는 중앙 수익금 대장 관리
- **권한 제어 시스템**: 낙찰 시 HR API 연동을 통한 연차 개수 즉시 증액 및 속성(Flag) 분리를 통한 차감 우선순위 제어
- **수익 배당 시스템**: 연말에 에스크로 수익금을 지분에 따라 복지카드 한도로 정산

> **[제외 범위]** 직원 간 직접 거래(P2P) 및 실제 현금(KRW) 결제는 법적 규제에 의해 시스템 범위에서 영구 배제한다.

### 1.3. 정의, 약어 및 약칭

| 용어 | 정의 |
|---|---|
| **B2E** (Business to Employee) | 회사가 매물을 발행하고 중개하는 합법적 거래 구조 |
| **연차 1일권** | 특정 날짜에 종속되지 않고 사내 그룹웨어에서 자유롭게 기안할 수 있는 "연차 잔여 개수 1개"를 의미하는 본 시스템의 단일 경매 매물 |
| **지분 (Stake)** | **판매자**가 공용 풀에 기여한 연차 일수의 비율 (연말 배당금 정산의 기준) |
| **휴가 속성 플래그 (Leave Type)** | `REGULAR` (법정 연차 / 수당 대상 / 공용 풀 전환 대상) <br> `AUCTION` (경매 낙찰 연차 / 연말 소멸 / 수당 제외 / 최우선 차감) <br> `EVENT` (사내 포상 연차 / 연말 소멸 / 수당 제외) |

### 1.4. 참조 (References)

## 2. 전체 설명

### 2.1. 제품 관점

본 시스템은 사내 그룹웨어와 **독립적인 MSA(Microservices Architecture)** 환경에서 동작하되, 다음 3가지 트랜잭션 단계에서만 외부 HR 시스템의 REST API와 연동된다.

1. 사용자 인증 (SSO)
2. 연차 개수 부여
3. 복지카드 한도 증액

회사의 자본 선투입 없이 시스템 내의 포인트 순환만으로 작동하는 **"무자본 중개 플랫폼"**이다.

### 2.2. 핵심 비즈니스 로직 (The 3-Way Win Model)

1. **[회사 / HR]** 연차 매입을 위한 예산 선투입 없음 → 에스크로 잔고 내에서만 배당하므로 **재무 리스크 0%**. 비정규 연차(`AUCTION`, `EVENT`)는 수당 정산에서 배제되어 예산 누수 없음.
2. **[판매자 (Seller)]** *— 주로 고연차 집단* — 연차 촉진제도로 소멸될 뻔했던 0원짜리 연차를 공용 풀에 기여하여 연말 배당금(복지 한도) 수령 → **100% 이득**.
3. **[구매자 (Buyer)]** *— 주로 저연차 집단* — 복지 포인트를 소모하여 부족했던 연차 개수를 확보하고 원하는 시기에 휴식.

> 💡 **역할 구분 주의**: "판매자", "구매자"는 **행위 기반 역할**로, 동일 직원이 두 역할을 동시에 수행 가능. 고연차/저연차는 예비군의 성향 설명일 뿐 근속연수로 시스템 권한이 결정되지 않음. 상세는 [용어집 F-1/F-2](glossary.md#f-참여자-actor) 참조.

### 2.3. 제약조건 (Constraints)

- **재무적 제약**: 어떠한 경우에도 에스크로에 누적된 잔액을 **1포인트라도 초과**하여 배당금을 지급할 수 없다.
- **데이터 무결성 제약**: 경매 낙찰 과정(포인트 차감 → 에스크로 증가 → HR API 호출)은 반드시 **단일 데이터베이스 트랜잭션**으로 처리되어야 한다.

## 3. 구체적 요구사항

### 3.1. 외부 인터페이스 요구사항

#### 1) 연차 권한 부여 API (HR 연동)

낙찰 확정 시 사내 HR 시스템으로 호출.

```http
POST /api/hr/leave
Content-Type: application/json

{
  "empId": "사번",
  "amount": 1,
  "type": "AUCTION"
}
```

#### 2) 배당금 지급 API (HR 연동)

연말 배당 시 호출.

```http
POST /api/hr/welfare
Content-Type: application/json

{
  "empId": "사번",
  "amount": "배당포인트"
}
```

### 3.2. 기능적 요구사항 명세표

| Req ID | 기능명 | 상세 요구사항 | 예외 및 오류 처리 |
|---|---|---|---|
| **FR-1.1** | 연말 정산 및 공용 풀 생성 | 매년 12월 31일 23:59 스케줄러가 `REGULAR` 미사용 연차를 취합하여 익년도 "연차 1일권" 매물 생성, 기여자 지분(Stake)을 대장에 기록 | 정산 대상 연차 0개 사용자 스킵 / 연차 반납 시 포인트 선지급 **절대 불가** |
| **FR-2.1** | 실시간 입찰 진행 | 잔여 포인트 내 입찰, 상위 입찰 발생 시 기존 최고가 입찰자에게 WebSocket 알림 발송 | 포인트 부족 시 입찰 버튼 비활성화 / **[제약]** Redis 분산 락으로 동시성 제어 필수 |
| **FR-2.2** | 낙찰 및 에스크로 적립 | 경매 마감 시 최종 낙찰자 포인트 차감, 시스템 에스크로 중앙 대장에 (+) 누적 적립 | 트랜잭션 실패 시 전체 롤백 / 관리자 채널(Slack 등) Critical 알림 발송 |
| **FR-2.3** | HR 휴가 권한 부여 | 낙찰 확정 즉시 HR API POST 호출하여 낙찰자 잔여 연차를 `AUCTION` 속성으로 +1 증가 | FR-2.2와 동일 DB 트랜잭션 내 처리 / API 장애 시 Message Queue 적재 후 재시도 |
| **FR-3.1** | 휴가 차감 우선순위 제어 | 휴가 사용 승인 시 `AUCTION → EVENT → REGULAR` 순으로 최우선 차감 | 1순위 잔고 0일 경우 다음 순위 폴백 / 일반 연차(`REGULAR`)는 최후 차감 |
| **FR-4.1** | 연말 배당금 정산 | 매년 12월 31일, 에스크로 총수익금을 기여자 지분율(Stake)에 따라 분할 산정하여 복지카드 한도 증액 API 호출 | **[제약]** 에스크로 잔액 초과 배당 절대 불가 / 산정액 불일치 시 배치 중단 및 수동 감사 대기 |
| **FR-4.2** | 잔여 재고 활용 및 청산 | 유찰된 1일권 재고는 사내 포상(`EVENT`)으로 수동 지급 지원, 당해 연도 종료 시 남은 재고 영구 삭제 | 수동 지급 시 에스크로 금액 변동 없음 / **[제약]** 재고의 익년 이월 회계상 원천 차단 |

### 3.3. 비기능적 요구사항 (Non-Functional Requirements)

#### [NFR-1] 동시성 제어 (Concurrency)

마감 직전 다수 사용자의 동시 입찰 시 발생하는 경쟁 상태(Race Condition)를 제어하기 위해 **Redis 기반의 분산 락(Distributed Lock)**을 필수 적용하여 단일 입찰가 정합성을 **100% 보장**해야 한다.

#### [NFR-2] 재무 정합성 및 감사 추적성 (Auditability)

시스템은 다음 공식을 보장해야 한다:

> **[연말 총 배당금 지급액] = [1년간 에스크로에 모인 포인트 총합]**

포인트 및 연차의 모든 변동은 **포인트 거래 대장(Transaction Log) 테이블에 Insert-Only 방식으로 영구 기록**되어야 한다 (수정/삭제 불가).

### 3.4. 논리적 데이터베이스 요구사항

본 시스템은 무결성 유지를 위해 **관계형 데이터베이스(PostgreSQL 권장)**를 사용하며, 핵심 엔티티와 제약조건은 다음과 같다.

#### 3.4.1. 핵심 엔티티 관계도 (ERD)

→ [ERD 상세 문서](../03_design/erd.md) 및 [UML 클래스 다이어그램](../03_design/UML.md#-클래스-다이어그램-class-diagram) 참조.

#### 3.4.2 데이터 무결성 제약조건

- **[DB-RULE-1] 대장 불변의 법칙**: `POINT_TRANSACTION_LOG` 테이블은 추적을 위해 오직 INSERT만 허용되며 UPDATE, DELETE는 시스템적으로 차단된다.
- **[DB-RULE-2] 연도별 정산 파티셔닝**: `LEAVE_BALANCE`는 `year` 기준으로 관리되며, 12월 31일 배치 시 `AUCTION`, `EVENT` 속성의 이전 연도 데이터는 자동 소멸(Soft Delete) 처리된다.
- **[DB-RULE-3] 외래키 및 잔액 제약**: 낙찰 확정 시 `AUCTION.winner_id`의 `USER.current_point`는 반드시 `highest_bid` 이상이어야만 트랜잭션이 Commit 된다.

### 3.5. 사용자 인터페이스 요구사항

- **실시간 렌더링**: 경매 마감 타이머와 실시간 최고가는 **WebSocket**을 통해 화면 새로고침 없이 동적으로 렌더링되어야 한다.
- **가시성**: 사용자는 자신의 지분율(Stake)에 따른 **"올해 예상 연말 배당금"**을 대시보드에서 상시 조회할 수 있어야 한다.
