# 1. 프로젝트 개요

## 해결하려는 문제 — 사내 "연차 양극화"
- 고연차 직원: 업무 과다로 연차를 다 못 써서 연말에 미사용 연차가 소멸(0원)되거나 수당으로만 정산된다.
- 저연차 직원: 법정 연차가 적어 휴가가 부족하다.
- 양쪽 모두 손해인 현상이 매년 반복된다.

## 폐기한 단순 아이디어
- "직원끼리 연차를 직접 사고팔자"(P2콘)는 근로기준법 제60·61조상 **휴식권 매매 금지**에 걸려 위법이다.
- 따라서 개인 간 직접 거래(P2콘)와 현금 결제는 설계에서 **영구 배제**했다.
- 진짜 도전 과제: **합법적이면서 회사 재무 리스크 0%**인 매칭 구조를 만드는 것.

## 해결책 — B2E Escrow & Dividend
회사가 중개하고(B2E), 대가는 사내 복지 콘, 정산은 연말 일괄 배당.
- 거래 주체: 직원↔직원(P2콘) → **회사가 중개(B2E)**
- 재화: 현금 → **사내 복지 콘**
- 정산: 즉시 → **연말 일괄 배당**
- 매물: 날짜 종속 연차 → **"연차 1일권"**(날짜와 무관한 개수)

핵심 용어 **수익적립금**: 구매자가 입찰로 낸 콘을 회사가 마음대로 쓰지 못하게 따로 모아두는 공동의 돈(중고거래 안전결제와 같은 개념). 연말에 이 모인 돈만큼만 판매자에게 배당한다. (영문 아키텍처명에선 Escrow)

## 사용자(액터) — 행위 기반 역할
- 직원(Employee): 모든 역할의 기반. 동일 직원이 시점별로 여러 역할을 동시에 수행 가능.
- 판매자(Seller): 공용 풀에 연차를 기여하고 연말 배당을 받는다.
- 구매자(Buyer): 경매에 입찰하고 낙찰 시 연차를 얻는다.
- 관리자(Admin): 유찰 재고 지급 등 운영.
- 외부: 스케줄러(12/31 배치), HR/ezpass 시스템.

## 3-Way Win (참여자별 가치)
- 회사/HR: 예산 선투입 0원, 재무 리스크 0%. 비정규 연차(AUCTION/EVENT)는 수당 정산에서 제외돼 예산 누수 없음.
- 판매자(주로 고연차): 소멸될 0원짜리 연차 → 연말 배당금(복지 한도) 수령.
- 구매자(주로 저연차): 콘으로 부족한 연차를 확보하고 원하는 시기에 휴식.

## 핵심 기능 — 5개 하위 시스템
1. 자산화: 연말에 미사용 연차를 모아 "연차 1일권" 매물과 기여자 지분(Stake)을 만든다.
2. 경매: 복지 콘으로 실시간 최고가 입찰·낙찰(WebSocket).
3. 수익적립금: 낙찰 대금을 모으는 중앙 대장.
4. 권한 제어: 낙찰자에게 연차 +1 부여 + 속성 플래그(REGULAR/AUCTION/EVENT) 분리.
5. 수익 배당: 연말에 Stake 비율대로 복지카드 한도로 정산.

# 2. 요구사항 분석

## 기능 요구사항(FR)과 우선순위(MoSCoW)
- (Must) FR-2.1 실시간 입찰: 콘 한도 내 입찰, 상위 입찰 시 WebSocket 알림.
- (Must) FR-2.2 낙찰 및 수익적립금 적립: 콘 차감 → 중앙 대장 누적.
- (Must) FR-2.3 휴가 권한 부여: 낙찰 시 AUCTION 연차 +1, 단일 트랜잭션.
- (Must) FR-3.1 차감 우선순위: AUCTION → EVENT → REGULAR(사용자 선택 불가).
- (Must) FR-1.1 연말 정산·공용 풀 생성: REGULAR 미사용 수집 → 매물 + Stake.
- (Must) FR-4.1 연말 배당금 정산: 수익적립금 → Stake 비례 복지 한도.
- (Should) FR-5.2 잔액·거래내역 조회(본인/관리자 전체).
- (Should) FR-5.1 관리자 콘 적립(CREDIT_ADMIN, 사유 필수).
- (Could) FR-4.2 유찰 재고 EVENT 수동 지급 + 연말 청산.

## 비기능 요구사항(NFR) — "사내 금융급 정합성"
- NFR-1 동시성: 마감 직전 동시 입찰의 경쟁 상태 0건. SQLite write 락(lockAuction)으로 같은 경매 입찰을 직렬화.
- NFR-2 재무 정합성·감사 추적: 배당 오차 0원. 거래 기록을 Insert-Only로 영구 보관하고, 통화별로 등식 검증.
- 보안: 중앙 인증(ezpass) 위임 + JWT, 권한 분리(EMPLOYEE/ADMIN).
- 실시간: 입찰가·타이머를 WebSocket으로 즉시 반영.

재무 정합성 불변식(NFR-2 핵심): `Σ(BID+WIN) − Σ(REFUND+DIVIDEND) = 수익적립금 잔액` (통화별). 즉 총 배당 = 수익적립금 총액, 1콘도 초과 불가.

## 절대 불변식(DB-RULE) — 깨지면 안 되는 규칙
1. 거래 기록 불변(Insert-Only): 거래 원장은 INSERT만, UPDATE/DELETE는 DB 트리거로 차단. 환불도 새 보정 INSERT.
2. 3-Flag 분리: REGULAR(수당 O) / AUCTION(수당 X) / EVENT(수당 X) → 이중 보상(Double-Dipping) 사고 방지.
3. 차감 우선순위 강제: AUCTION → EVENT → REGULAR.
4. 수익적립금 상한: 통화별 배당 합 = 수익적립금 잔액(초과 금지).

이 시스템은 단순 연차 게시판이 아니라 사내 복지 예산·HR 정산이 직결된 플랫폼이므로 무결성이 1순위 요구사항이다.

# 3. UML 모델링 (텍스트 설명 — 실제 이미지는 별도 첨부 필요)

> 실제 다이어그램 이미지 8장은 `docs/03_design/uml/`에 PNG로 있다(usecase, class, sequence, state, activity-year-end, component-hexagonal, object-integrity, activity-deduction). 슬라이드에는 이 PNG를 직접 삽입해야 한다. 아래는 각 다이어그램이 보여주는 내용의 텍스트 설명이다.

## 유스케이스 다이어그램
6 액터(일반화 3건: 직원←판매자/구매자/관리자) · 15 유스케이스. 입찰은 실시간 알림을 «include», 낙찰은 HR 연차 부여를 «include». 콘 부족·HR 재시도·유찰은 조건부 «extend» 3건. HR 시스템은 호출을 받기만 하는 Secondary 액터.

## 클래스 다이어그램
13 클래스·5 열거형. LeaveBalance가 REGULAR/AUCTION/EVENT 3종으로 일반화되고 각자 연말 소멸 동작을 다형적으로 구현. HRApiClient는 인터페이스↔구현체 실체화. LedgerEntry(거래 원장)는 «Insert-Only». Wallet·Escrow·Stake가 핵심 엔티티. 가시성·다중도·연관/집합/컴포지션 모두 표기.

## 순차 다이어그램 — 입찰→낙찰→연차 부여
write 락 선점 후 단일 트랜잭션 안에서 wallet 차감·최고가 갱신·거래 기록 INSERT를 묶는다. 동기·비동기·반환 메시지와 alt/par/opt 프레임 구분. 미래 그룹웨어 연동 시의 Outbox 분기까지 표현.

## 상태 다이어그램 — Auction 객체 생애주기
6 상태. CREATED→OPEN→CLOSED에서 choice로 낙찰/유찰 분기, 이후 AWARDED 또는 UNSOLD를 거쳐 연말에 EXPIRED로 수렴. 각 상태에 entry/do/exit 액션.

## 활동 다이어그램 — 연말 배치(수익적립금→배당) [핵심]
1년간 쌓인 수익적립금을 연말에 처리하는 전체 흐름. ① REGULAR 미사용 취합 → ② 기여분>0이면 Stake 계산 → ③ 수익적립금 총액 집계 후 **정합성 등식 검증** → 통과해야만 배당 산정(floor 내림 + 나머지는 Stake 1위에 가산) → ④ 복지카드 한도 증액 → ⑤ 다음 해 매물 생성과 이전연도 연차 소멸을 병렬 처리. 검증 실패 시 배당을 실행하지 않고 배치를 중단해 수동 감사로 넘긴다(재무 사고 방지).

## 컴포넌트 다이어그램 — Hexagonal Ports & Adapters
계층: interfaces(Inbound) → application(Use Cases) → domain(Core, 외부 의존 0) ← ports(«interface») ← adapters. 어댑터가 포트를 실체화(implements). 의존은 항상 안쪽으로. 5개 바운디드 컨텍스트(Auction·Wallet·Leave·LeavePool·Dividend).

## 객체 다이어그램 — 정합성 스냅샷
실제 숫자로 NFR-2 증명: 경매 A·B 낙찰로 수익적립금 10000. 기여자 박(지분 2/3)·최(1/3). 배당 floor 6666/3333, 나머지 1은 Stake 1위(박)에게 → 6667/3333. 합 10000 = 잔액. 1콘도 초과/누락 없음.

## UML 일관성
이 다이어그램들은 하나의 도메인 모델(요구사항·ERD·설계 결정)에서 파생된다. 유스케이스의 기능이 클래스 구조로, 그 흐름이 순차로, Auction 상태 필드가 상태기계로, 1년 흐름이 활동으로 이어진다. 서로 모순 없이 추적 가능한 것이 "정확성"의 근거다(UML 7종 표기법).

# 4. 소프트웨어 아키텍처

## 구조 선택 — Hexagonal (Ports & Adapters), 근거
- 이 시스템의 본질은 도메인 규칙(수익적립금 등식·3-Flag·차감 우선순위)이라, 도메인을 프레임워크/DB/외부 API로부터 격리해야 규칙이 흔들리지 않는다.
- 외부 자원(화폐·휴가·HR 연동 시점)이 미정/교체 가능 → 포트 인터페이스 뒤로 추상화하면 코어 무수정으로 어댑터만 교체(개방-폐쇄 원칙).
- 도메인 코어가 외부 의존 0이라 2인 팀이 단위 테스트하기 좋다.
- 단순 레이어드(MVC)는 도메인이 ORM/프레임워크에 종속돼 금융급 불변식 보호에 취약하여 기각.

## 블록 다이어그램(논리 아키텍처)
- 클라이언트(직원 웹 UI·관리자 UI, React) → API Gateway/인증 미들웨어(JWT) → 서비스(AuthService·AuctionService·DividendService·AdminService) → 저장소(SQLite: 거래 원장+지갑+연차) / Outbox(HR 호출 큐, 현재 dormant) / 외부(ezpass 인증, HR 배당 출금).
- 논리적으로는 MSA로 분리, 물리적으로는 단일 배포로 시작. 헥사고날 경계가 미래 마이크로서비스 분리선.

## 단일 트랜잭션
낙찰 정산은 wallet 차감 + 수익적립금 적립 + 거래 기록 INSERT + 연차 부여가 단일 DB 트랜잭션으로 All-or-Nothing. 입찰 경로엔 외부 호출이 0이라 분산 트랜잭션 문제가 없다.

# 5. 적용 디자인 패턴 (GoF — 어디에·왜)

## Adapter (구조 패턴)
- 어디에: ports ↔ adapters. BiddingCurrency 포트를 WelfarePointProvider가, 인증을 ezpass 어댑터가, LeaveGrantPort를 InternalLeaveAdapter가 구현.
- 왜: 외부 자원(화폐·휴가·인증)을 갈아끼워도 도메인 코어 무수정(OCP). 헥사고날의 심장.

## Observer (행위 패턴)
- 어디에: 도메인 이벤트. 입찰/낙찰 유스케이스가 이벤트를 발행만 하고, WebSocket 브로드캐스트·메트릭·감사 로그는 구독자로 분리. NestJS EventEmitter로 구현.
- 왜: 새 부수효과를 추가해도 유스케이스를 수정하지 않음(횡단 관심사 분리).

## State (행위 패턴) — 설계 + 정직한 트레이드오프
- 어디에: Auction 상태 전이.
- 설계 의도: 상태를 객체로 나눠 if(status==…) 분기를 없앤다.
- 구현 현실: 2인·단일 SQLite 범위에 맞춰 enum + guard 메서드로 단순화(스코프 컷으로 문서화). 단, 어댑터/유스케이스가 status로 분기하지 않고 placeBid()/settle() 호출 후 예외만 처리한다는 패턴의 의도는 유지.
- 왜 중요: 패턴을 왜 쓰는지뿐 아니라 "왜 이 범위에선 단순화했는지"까지 근거를 남긴 것이 적용의 타당성.

## 보조 패턴
- Strategy: 통화 Provider 교체 가능성. Repository·Unit of Work: 영속성/트랜잭션 경계로 도메인·DB 분리.

# 6. 역할 분담 + 14주차 구현 계획

## 현재 구현 현황(13주차)
설계에 그치지 않고 헥사고날 골격과 핵심 경로까지 구현 착수: 도메인 코어·경매 유스케이스 6종·ezpass 인증 통합·SQLite·프론트 13개 화면 골격. 14주차 핵심은 연말 배치·배당.

## 역할 분담(2인, 주담당+부담당 크로스 리뷰)
- 김기철: 경매/입찰·낙찰, 동시성, 연말 배치, 인프라(CI/CD).
- 오지석: 인증, 연차(차감 우선순위), 배당(Stake), 관리자 API, 프론트엔드 주도.
- 기술 선택·신규 설계 결정은 양자 합의 원칙.

## 14주차 마일스톤
- M-A 경매 핵심 마감: 동시 입찰 부하 테스트에서 입찰가 정합 100%.
- M-B 연차·차감: AUCTION→EVENT→REGULAR 통합 테스트 통과.
- M-C 연말 배치·배당: 시드 데이터로 "배당 합 = 수익적립금" 검증(NFR-2).
- M-D 관리자·실시간: 유찰 EVENT 지급·WebSocket 동작.
- M-E 통합·시연: E2E + 데모 시나리오대로 무중단 시연.
- 리스크: 동시성(database is locked), 일학습병행 일정, 프론트-백 스펙 불일치 → 각각 대응책 보유.

# 핵심 메시지 (발표 처음과 끝에 반복)
직원끼리 연차를 사고파는 건 위법이라, 회사가 중개하고 연말에 배당하는 B2E 수익적립금(Escrow) 모델로 풀었고, 그 무결성을 불변식·헥사고날·GoF 패턴으로 설계했다. 설계에 그치지 않고 핵심을 구현해 14주차 시연으로 잇는다.

# 예상 질문 & 답변 (발표 Q&A 대비)

Q. 직원끼리 거래가 불법인데 경매는 합법인가? — 거래 주체가 다르다. 회사가 매물을 발행·중개(B2E)하고 직원은 휴식권을 직접 매매하지 않는다. 대가도 복지 콘이고 회사는 예산을 선투입하지 않는다. P2콘 경로는 어디에도 만들지 않는다.

Q. 회사가 돈을 안 쓰는데 판매자는 무슨 돈으로 배당받나? — 구매자들이 입찰로 낸 콘이 수익적립금에 쌓이고, 그 잔액만큼만 연말에 판매자 지분율대로 나눈다. 재원이 곧 구매자 지출이라 회사 부담이 0.

Q. 배당이 정확히 떨어지나? 나머지는? — 지분 비례로 내림(floor)한 뒤 나누어떨어지지 않는 나머지를 지분 1위에게 몰아준다. 그래서 합이 정확히 수익적립금 잔액이 된다. 동률이면 user_id 오름차순.

Q. 동시 입찰이 몰리면 입찰가가 꼬이지 않나? — 같은 경매에 대해 SQLite write 락으로 직렬화한다. 트랜잭션이 짧고 커밋/롤백 시 자동 해제. 별도 락 인프라(Redis)는 단일 인스턴스라 불필요해서 스코프 컷.

Q. State 패턴을 실제로 구현했나? — 설계는 6상태 State 패턴, 구현은 enum+guard로 단순화(스코프 컷). 2인·단일 SQLite 범위에서 상태별 메서드가 크게 갈리지 않아서다. status 분기 금지라는 패턴 의도는 유지했고, 분기가 커지면 객체로 승격하도록 문서화.

Q. 왜 MSA라면서 단일 배포인가? — wallet·수익적립금·stake를 공유하기 때문에 처음부터 물리 분리하면 분산 트랜잭션 비용만 크다. 논리적 경계만 먼저 긋고 헥사고날 경계가 미래 분리선이 되게 했다.

Q. HR/그룹웨어 연동은? — 학교 범위에선 휴가 부여를 내부 어댑터로 처리해 단일 트랜잭션으로 닫았다. 실 도입 시 그룹웨어 어댑터로 교체하면 Outbox 재시도·멱등 키 경로가 활성화된다.

Q. 인증은 직접 만들었나? — 사내 중앙 인증 ezpass에 위임하고 JWT로 세션을 관리한다. 신원·조직·권한은 ezpass, 연차·지갑 잔액은 우리 시스템이 마스터.

Q. 관리자가 자기 경매에 입찰하면 이해상충 아닌가? — 인지된 리스크. 입찰은 허용하되 관리자 입찰을 거래 기록에 별도 플래그로 남겨 권한과 감사에서 분리 추적.

Q. 2인 팀인데 14주차 안에 되나? — 핵심 경로는 이미 구현돼 있어 14주차는 연말 배치·배당과 통합 시연에 집중. 마일스톤별 완료 기준을 정량으로 잡았다.
