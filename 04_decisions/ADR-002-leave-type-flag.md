# ADR-002: 휴가 속성 3-flag 분리 (REGULAR / AUCTION / EVENT)

- **상태**: ✅ Accepted
- **결정일**: 2026-04-10
- **결정자**: 타임소프트콘

## 컨텍스트

낙찰 시 직원의 연차 개수를 HR 시스템에서 증가시켜야 한다. 이 증가된 연차를 기존 법정 연차와 어떻게 구분해서 다룰 것인지가 회계/재무 정합성의 핵심이었다.

## 초기 아이디어 (Rejected)

> "경매에서 낙찰받으면 사내 API를 호출해서 그냥 해당 직원의 **잔여 연차 개수 + 1** 처리한다."

즉, 일반 연차와 경매 연차를 **동일하게 취급**.

## 회피한 리스크

💣 **이중 보상 사고 (Double Dipping)**

낙찰받은 연차를 직원이 연말까지 **안 썼을 경우**:

1. 직원은 복지 포인트로 연차를 **구매**했음 (포인트 차감)
2. HR 시스템은 이를 "미사용 일반 연차"로 인식
3. 연말 수당 정산에서 **현금 연차수당을 또 지급**
4. **결과**: 직원은 포인트 내고 산 연차를 → 현금으로 환급받는 꼴

이는 **사내 재무팀이 뒤집어질 대형 회계 사고**이며, 어뷰징 수단으로 악용될 수 있음.

추가 리스크:
- 경매 낙찰 연차의 **이월 발생** → 영구 누적되어 시스템 복잡도 증가
- 공용 풀 재투입 시 **무한 순환 구조** 발생 가능

## 결정

`LEAVE_BALANCE` 테이블에 `leave_type` enum 컬럼을 도입하여 **3가지 속성으로 완벽 분리**한다.

| Flag | 의미 | 연말 수당 | 연말 처리 | 차감 순위 |
|---|---|---|---|---|
| `REGULAR` | 법정 연차 | ✅ 대상 | 공용 풀 전환 | 3순위 |
| `AUCTION` | 경매 낙찰 | ❌ 제외 | **영구 소멸** | **1순위** |
| `EVENT` | 사내 포상/이벤트 | ❌ 제외 | **영구 소멸** | 2순위 |

### HR API 연동 시

```json
POST /api/hr/leave
{ "empId": "...", "amount": 1, "type": "AUCTION" }
```

→ HR 시스템도 이 플래그를 기반으로 **연말 수당 정산에서 AUCTION/EVENT를 제외**해야 함.

## 결과 및 트레이드오프

### ✅ 긍정적 결과
- **이중 보상 원천 차단** — 재무팀/HR 정합성 보장
- **명확한 속성 구분** — 감사 추적성 확보
- **연말 이월 자동 차단** — AUCTION/EVENT는 무조건 소멸

### ⚠️ 트레이드오프
- **HR 시스템 측 개발 부담** — HR API가 type 필드를 인식하고 수당 정산에서 분리 처리해야 함 → 사내 HR팀과의 긴밀한 협의 필요
- **테이블 복잡도 소폭 증가** — `(user_id, year, leave_type)` 복합 UNIQUE 키
- **3-flag의 유지 비용** — 향후 `SPECIAL` 등 플래그 추가 요구 시 enum 변경 + 전체 차감 로직 수정

### 🛡️ 제약
- HR 시스템이 **type 필드를 보존·인식하지 않으면** 전체 모델이 붕괴 → HR API 스펙 협의 필수
- 직원이 쓰다 만 `AUCTION` 연차는 **연말에 포인트 환불 없이 소멸** — UI에 반드시 고지 필요
